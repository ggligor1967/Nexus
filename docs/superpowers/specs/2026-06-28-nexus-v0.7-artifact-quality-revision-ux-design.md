## Slice 1 — Better Markdown export

### Goal

Produce a Markdown export that is both **human-readable** and **machine-auditable**, rendered deterministically from the validated `plan_json`. `exportablePlanMarkdown` is **not** the source of truth and does not appear in the main output.

### Renderer change — `src/lib/export/markdown.ts`

The renderer must stay deterministic so its output is byte-assertable in E2E. Metadata that varies (export time, model) is **threaded in**, never read inside the renderer (no `new Date()`, no `process.env` reads in the function). All metadata values originate from **persisted** columns (`ai_plan_runs.completed_at`, `ai_plan_runs.model_name`) and the validated plan; the renderer never synthesizes a value.

New signature:

```ts
export interface ExportMeta {
  exportedAt: string;   // ai_plan_runs.completed_at (persisted; guaranteed non-null for a completed run)
  modelName: string;    // ai_plan_runs.model_name (persisted)
  riskLevel: RiskLevel; // plan.ethicalRiskReport.overallRiskLevel
}

export function planToMarkdown(
  projectName: string,
  plan: NexusPlan,
  meta: ExportMeta
): string
```

`sanitizeFilename` is unchanged.

### Frontmatter safety (hard constraint)

All values interpolated into the YAML frontmatter must be **safely escaped** so a project title (or any string) containing `"`, `:`, `\`, `#`, or a newline cannot break the document or inject YAML keys. A single local helper handles this:

```ts
// Always emit a double-quoted YAML scalar with \ and " escaped and newlines flattened.
function yamlString(value: string): string {
  const escaped = value
    .replace(/\\/g, "\\\\")
    .replace(/"/g, '\\"')
    .replace(/\r?\n/g, " ");
  return `"${escaped}"`;
}
```

Every frontmatter value (`title`, `exported_at`, `risk_level`, `model`, and the constant `format`/`source`) is emitted through `yamlString`. `risk_level` and `model` come from a constrained enum/persisted set, but are escaped uniformly for consistency. The `# <projectName>` H1 heading is plain Markdown (not YAML) and needs no YAML escaping, but newlines in the title are flattened there too.

### Output structure

```markdown
---
title: "<projectName>"
exported_at: "<meta.exportedAt>"
risk_level: "<meta.riskLevel>"
model: "<meta.modelName>"
format: "nexus-markdown-v1"
source: "validated_plan_json"
---

# <projectName>

> **Risk Level: <RISK_LEVEL_UPPERCASE>**
> Review safeguards and red lines before marking this plan build-ready.

## 1. Product Thesis
## 2. Deconstruction
### 2.1 User Problems
### 2.2 Target Audience
### 2.3 Jobs To Be Done
### 2.4 Constraints          # NEW — currently missing from export
### 2.5 Assumptions
### 2.6 Unknowns
### 2.7 Success Metrics
## 3. Product Strategy        # Positioning / Differentiator / Core Value Proposition / MVP Boundary / Non-Goals
## 4. Prototype Options
### Option 1 — <title>        # numbered
## 5. Ethical Risk Report
## 6. Roadmap
## 7. Build-Ready Checklist
```

Notes:
- The Constraints subsection renders `plan.deconstruction.constraints`.
- Risk callout uses the uppercase `overallRiskLevel`.
- The build-ready checklist remains a static checklist but is retained and clarified; it is not derived from AI output.

### Route change — `src/app/api/projects/[id]/export/markdown/route.ts`

After loading the latest **completed** run (existing query), build `ExportMeta` purely from persisted columns and the validated plan, then pass it in:

```ts
const meta: ExportMeta = {
  exportedAt: latestRun.completed_at,   // persisted; a completed run always has completed_at
  modelName: latestRun.model_name,      // persisted
  riskLevel: validatedPlan.ethicalRiskReport.overallRiskLevel
};
const markdown = planToMarkdown(project.title, validatedPlan, meta);
```

No `new Date()` anywhere in the export path: the generate-plan route sets `completed_at` when it marks a run `completed`, so `latestRun.completed_at` is a guaranteed-present persisted value for any run the export route accepts (it only loads runs with `status = "completed"`). The existing `exported_plans` insert and `projects.status = "exported"` update are unchanged.

### Slice 1 acceptance

```text
- Markdown contains YAML frontmatter with title, exported_at, risk_level, model, format, source
- Frontmatter values are escaped: a title containing `"` or a newline produces valid YAML (no broken doc, no injected keys)
- Markdown contains a Constraints section
- Markdown contains a risk-level callout blockquote
- Markdown contains at least one numbered prototype option ("### Option 1 —")
- exported_plans row is still created
- planToMarkdown is pure: same (projectName, plan, meta) => identical string
- typecheck / build / E2E green
```

## Slice 2 — Revision history

### Semantics (model A)

```text
First valid generation:
  - creates a completed ai_plan_runs row
  - creates NO revision row (no prior completed plan exists)

Second and later valid generations:
  - previous_snapshot = previous completed plan_json
  - new_snapshot      = newly validated plan_json
  - ai_plan_run_id    = new completed ai_plan_runs.id
  - revision_note     = "Plan regenerated"

No revision is written for: invalid_json, failed generation, schema-validation failure.
```

### Write path — `src/app/api/projects/[id]/generate-plan/route.ts`

1. After auth + ownership checks, **before** creating the new run, query the latest completed run's `plan_json` and hold it as `previousPlan` (may be null).
2. Proceed with the existing flow: insert `concept_inputs`, create the new `ai_plan_runs` row (status `running`, via admin client), generate, JSON-parse, Zod-validate, update the run to `completed`.
3. In the **success branch only**, if `previousPlan` is non-null, insert one `revisions` row via the **admin (service-role) client**:

```ts
{
  project_id: projectId,
  ai_plan_run_id: run.id,
  previous_snapshot: previousPlan,
  new_snapshot: validatedPlan,
  revision_note: "Plan regenerated"
}
```

4. **Failure mode:** a failed revision insert is logged and swallowed — it must **not** convert an already-earned 200 generation into a 500. The run is completed and the response is returned regardless.

`previousPlan` is captured at the start (querying `status = "completed"`), so the just-created `running` row is never mistaken for the previous plan.

### Migration — `supabase/migrations/0003_restrict_revision_writes.sql`

Mirrors SEC-003 (migration `0002`) for `revisions`. Authenticated users keep SELECT on their own revisions; direct client INSERT/UPDATE is removed so writes go only through the service-role path.

```sql
-- v0.7: revision rows are server-written audit events.
-- Authenticated users may read revisions for their own projects, but direct
-- client INSERT/UPDATE is denied; writes go through service-role server code.
DROP POLICY IF EXISTS "Users can insert revisions for own projects" ON public.revisions;
DROP POLICY IF EXISTS "Users can update revisions for own projects" ON public.revisions;
```

Note: migration `0001` never created an UPDATE policy on `revisions`, so the second DROP is a harmless no-op kept for symmetry with the SEC-003 pattern. The SELECT policy ("Users can select revisions for own projects") is intentionally retained.

### Project page fix — `src/app/projects/[id]/page.tsx`

The page currently fetches `latestRun` by `created_at` with **any** status, then renders the plan only when `latestRun.status === "completed"`. Once the regenerate path (Decision 3) exists, a failed regeneration would set the latest run to `failed` and **wipe the still-good plan, ExportActions, and the revision panel**.

Fix the data fetch so display no longer conflates "latest run" with "latest completed run":

- `latestCompletedRun` (status `completed`, newest first) → drives `GeneratedPlanView`, `ExportActions`, `RevisionHistoryPanel`, and the regenerate affordance.
- `latestRun` (any status, newest first) → used **only** to show a failure banner above the plan when its status is `failed`.
- If `latestCompletedRun` is null → show `ConceptIntakeForm` (first-run state), plus the failure banner if the latest run failed.

This aligns the page with the export route and build-ready check, which already query the latest *completed* run.

### Completed-plan view composition

When `latestCompletedRun` exists, the project page renders:

```text
[failure banner if latestRun.status === "failed"]
GeneratedPlanView      (live, interactive — keeps the ethical acknowledgement controls)
ExportActions
RevisionHistoryPanel   (server-fetched revisions for this project, newest-first)
RegeneratePanel        (client component, see below)
```

`RevisionHistoryPanel` is presentational; the project page (server component) fetches the revisions list and passes it in. Each row shows: revision number, timestamp, `new_snapshot` risk level, revision note, short run id, and a "View snapshot" link to the detail route.

### Regenerate path — `RegeneratePanel` (new client component)

- Default: shows a "Regenerate plan" button (`data-testid="regenerate-plan-button"`).
- On click: reveals `ConceptIntakeForm`, **prefilled** from the latest `concept_inputs` row when available (the project page fetches it and passes initial values; the form gains optional `initialValues` props with current behavior as the default).
- Submit reuses the existing generate-plan flow; on success `ConceptIntakeForm` already calls `router.refresh()`, so the new plan and the new revision row appear.
- The button introduces **no** restore/branch/edit semantics — it only triggers a normal forward generation.

`ConceptIntakeForm` change: accept optional `initialValues` (raw concept, target users, platform, language, risk domain, output type). Absent props preserve today's defaults, so the first-run intake is unchanged.

### Snapshot detail route — `src/app/projects/[id]/revisions/[revisionId]/page.tsx` (new)

- Server component, `dynamic = "force-dynamic"`.
- Auth + ownership: load the revision joined to its project; if the project is not owned by the current user (or the revision does not exist), `notFound()` (404). RLS enforces this at the DB layer; the route also checks explicitly.
- Renders **both** snapshots, each via `GeneratedPlanView` in a new `readOnly` mode:

```text
# <project title> — Revision <n>
## Previous version   -> GeneratedPlanView(plan = previous_snapshot, readOnly)
## New version        -> GeneratedPlanView(plan = new_snapshot, readOnly)
[back link to /projects/[id]]
```

Rendering both makes "what changed" explicit and keeps the very first plan reachable — gen1 lives only in the earliest revision's `previous_snapshot`.

`GeneratedPlanView` change: add a `readOnly?: boolean` prop. When `readOnly`, the ethical risk content still renders, but the build-ready **acknowledgement controls** (checkbox, mark/clear buttons, PATCH calls) are suppressed. Implementation: thread `readOnly` to `EthicalRiskPanel`, which renders the report body but skips the interactive acknowledgement section when `readOnly` is true. `projectId` is still accepted but no mutation is possible in read-only mode.

### Slice 2 acceptance

```text
DB / server:
- first valid generation creates 0 revision rows
- second valid generation creates exactly 1 revision row
- previous_snapshot stores the old validated plan_json
- new_snapshot stores the new validated plan_json
- ai_plan_run_id references the new completed run
- invalid_json regeneration creates no revision row
- a failed revision insert does not fail the generation response

Security:
- authenticated client can SELECT its own revisions
- authenticated client direct INSERT on revisions is denied (RLS)
- another user cannot read this project's revisions
- another user cannot reach the snapshot detail route (404)

UI / UX:
- completed plan, export, and revision panel survive a failed regeneration (failure banner shown above them)
- "Regenerate plan" reveals a prefilled ConceptIntakeForm
- Revision History lists revisions newest-first with number, timestamp, risk level, note
- "View snapshot" opens the read-only detail route showing previous + new versions
- read-only snapshot view exposes no acknowledgement controls
```

### E2E plan (extends `tests/e2e/nexus-v05.spec.ts`)

To make snapshots differ observably (the fixture is deterministic per scenario), the two generations use **different** concepts:

```text
1. sign up / login as user A, create a project
2. gen1: submit a STANDARD concept via the UI -> plan renders, revision history empty
3. click "Regenerate plan" -> ConceptIntakeForm appears (prefilled)
4. gen2: select risk domain "surveillance" + submit a SURVEILLANCE concept (-> critical)
5. revision history shows exactly one revision (newest-first)
6. assert new_snapshot.overallRiskLevel !== previous_snapshot.overallRiskLevel
   (e.g. via admin client read of the revisions row, mirroring existing DB assertions)
7. "View snapshot" opens /projects/[id]/revisions/[revisionId]; both versions render read-only
8. client direct INSERT into revisions is DENIED (anon/user client)
9. invalid_json regeneration (submit a concept containing the token `invalid_json`,
   mirroring the existing PLAN-003 test — no env toggle mid-suite) creates no revision
10. RLS: user B cannot read user A revisions and gets 404 on the snapshot route
```

## Hard constraints (consolidated)

These are non-negotiable acceptance gates, restated in one place:

```text
Slice 1
- planToMarkdown must not call new Date() (or read process.env / any nondeterministic source)
- all export metadata comes from persisted values (ai_plan_runs.completed_at, .model_name) + validated plan
- frontmatter values must be safely escaped (yamlString) against ", :, \, newlines, key injection
- exportablePlanMarkdown is excluded from the main output

Slice 2
- successful regeneration with a prior completed plan creates exactly one revision
- first generation creates no revision
- invalid_json / failed / schema-invalid generation creates no revision
- revisions are server-written only; migration 0003 removes client INSERT/UPDATE; SELECT-own retained
- a failed revision insert is logged but non-fatal (never converts a valid 200 into a 500)
- the snapshot route returns 404 for a non-owner (RLS + explicit ownership check)
- readOnly mode suppresses all mutation controls (no acknowledgement checkbox/buttons, no PATCH)
```
